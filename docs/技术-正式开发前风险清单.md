## 一、Apify 调用与缓存策略

### 问题 1：同一链接重复调用浪费额度

**原因：** 前端每次刷新或每次点击"分析"按钮，后端都发起新的 Apify Actor run。一次 run 耗时 20-40 秒，成本约 $0.03-0.05/次。当前免费额度仅剩 12 次。

**主流方案：** 后端缓存层 —— `offerId → 结果 JSON`，TTL 24 小时。首次调用 Apify，后续 24h 内读缓存直接返回。

**依据：** Apify 按次计费，不缓存 = 烧钱 + 慢。1688 批发价变动频率低（周级别），24h TTL 对数据新鲜度影响可接受。

---

### 问题 2：并发请求同一链接导致重复调用（缓存击穿）

**原因：** 用户 A 和用户 B 几乎同时贴同一个 1688 链接。A 的请求去调 Apify 了，但结果还没写入缓存；B 的请求查缓存为空，也去调 Apify。两次调用，一份数据。

**主流方案：** 请求合并（Request Coalescing）。Python 实现：用 `asyncio.Event` 或字典存 pending futures。第一个请求去调 Apify，后续相同 offerId 的请求挂到同一个 Promise/future 上等结果。

```python
# 伪代码示意
_pending: dict[str, asyncio.Task] = {}

async def fetch_product(offer_id: str):
    if offer_id in _pending:
        return await _pending[offer_id]  # 等第一个请求的结果
    if cached := await cache.get(offer_id):
        return cached
    task = asyncio.create_task(_call_apify_and_cache(offer_id))
    _pending[offer_id] = task
    try:
        return await task
    finally:
        _pending.pop(offer_id, None)
```

**依据：** 经典并发控制模式。不用分布式锁（杀鸡用牛刀），单机字典即可，和 24h 缓存配合使用。

---

### 问题 3：缓存何时失效？数据变旧了怎么办？

**原因：** 1688 价格、库存会变动。24h 旧数据可能不准。但自动刷新又浪费额度。

**主流方案：** TTL 24h 自动过期 + 手动「强制刷新」按钮。页面顶部显示数据抓取时间戳，用户觉得旧了点一下强制刷新。不自动刷新。

**依据：** 1688 批发价变动频率为周级别，24h 对绝大多数场景够用。库存变动可能更快，但卖家核心关注的是价格和供应商资质（更稳定）。手动刷新给用户控制权，不浪费额度。

---

### 问题 4：Apify 调用超时

**原因：** Apify Actor 冷启动 + 爬取有时超过 60 秒。不做超时处理 → 前端一直转圈 → 用户以为坏了。

**主流方案：**
- 后端超时设 90 秒
- 超时后：有缓存（即使过期）返回缓存 + "数据可能不是最新"警告；无缓存返回明确错误
- 前端轮询每 2 秒一次，超过 30 秒显示"正在获取，预计还需 X 秒"的进度提示

**依据：** Apify 官方建议 `call()` 不设 timeout，但生产环境必须有兜底。90 秒覆盖 95% 正常情况。进度提示参考 OpenAI/Midjourney 的异步轮询模式。

---

### 问题 5：图片 1688 CDN 在东南亚加载慢

**原因：** 1688 图片 CDN 在国内部署，越南/印尼用户直连加载 5-10 秒。

**主流方案：** 后端代理下载图片 → 转存本地或 OSS → 前端用代理后的 URL。V1 至少加 `<img loading="lazy">` + 低分辨率占位图。

**依据：** 东南亚 4G 占比高，大资源 = 高跳出率。Demo 里已有 `loading="lazy"`，正式版需更彻底方案。

---

### 问题 6：恶意刷新与高频调用导致成本失控

**原因：** V1 阶段 Apify 免费不限次使用。恶意刷新、脚本攻击、或用户高频使用（如每分钟搜多次不同商品），会导致 Apify 调用成本线性增长。Apify `zen-studio/1688-wholesale-scraper` 约 $0.03/次，单人日刷 50 次 = $1.5/天 = $45/月。

**主流方案：** 三层防御体系，用户无感：

| 层 | 方案 | 参数 | 防什么 | 实现位置 |
|----|------|------|--------|---------|
| **L1 前端** | 搜索按钮 loading 防重 | 点击后 disabled 3s | 手抖重复点 | `report.js` 搜索按钮 |
| **L2 缓存** | offer_id 内存缓存 | 30 分钟 TTL，500 条上限，LRU 淘汰 | 同一商品重复搜 | `domain/cache.py` |
| **L3 全局** | 滑动窗口全局限流 | 30 次/分钟（所有用户合计） | 脚本攻击兜底 | `middleware.py` 或 `routes/analyze.py` |
| **L4（暂不做）** | IP 日限硬限制 | 30 次/天/IP | 等用户多了再上 | — |

**L2 缓存实现：**

```python
# domain/cache.py
import time

_analysis_cache: dict[str, tuple[float, dict]] = {}  # {offer_id: (timestamp, data)}
MAX_SIZE = 500
TTL = 1800  # 30 分钟

def get_cached(offer_id: str) -> dict | None:
    entry = _analysis_cache.get(offer_id)
    if entry and time.time() - entry[0] < TTL:
        return entry[1]
    return None

def set_cache(offer_id: str, data: dict) -> None:
    _analysis_cache[offer_id] = (time.time(), data)
    # LRU 淘汰：超上限删最老的
    if len(_analysis_cache) > MAX_SIZE:
        oldest = min(_analysis_cache, key=lambda k: _analysis_cache[k][0])
        del _analysis_cache[oldest]
```

**L3 全局限流实现：**

```python
# 滑动窗口
from collections import deque

_global_timestamps: deque[float] = deque()
GLOBAL_LIMIT = 30   # 次
GLOBAL_WINDOW = 60  # 秒

def check_global_rate() -> bool:
    now = time.time()
    while _global_timestamps and now - _global_timestamps[0] > GLOBAL_WINDOW:
        _global_timestamps.popleft()
    if len(_global_timestamps) >= GLOBAL_LIMIT:
        return False  # 超限
    _global_timestamps.append(now)
    return True
```

**设计原则：**
- 三层防御对正常用户完全透明（缓存命中反而更快，按钮防重是标准 UX）
- V1 不设硬性次数限制——宁可多花 $20 也不要让用户碰到"今日次数用完"提示
- 等有真实用户数据后，再根据实际使用模式调整 V2 定价策略

**依据：** 智囊团 2026-07-21 讨论结论。V1 核心目标是验证产品价值，不是盈利。防刷措施确保成本可控即可，不过度设计。

## 二、判词引擎

**判词定义：** 页面中每个 Tab 顶部的总结性文字（如"源头旗舰·高回头率·可小单试水"），是页面的核心价值——把原始 JSON 翻译成卖家能秒懂的结论。

### 问题 1：判词覆盖率不足

**原因：** 规则引擎不可能穷举所有产品类型和数值组合。如果只写 10 条精确规则，大量产品落到"无判词可匹配"。

**主流方案：** 三层规则金字塔：

| 层级 | 策略 | 示例 | 目标覆盖率 |
|------|------|------|-----------|
| **L1 精确匹配** | 多条件 AND 组合，命中即出精确判词 | "好评率>95% AND 回头率>70% AND tpYear≥5" → "高信誉老店" | ~40% |
| **L2 区间匹配** | 单指标阈值分段，组合出通用判词 | "回头率 50-70%" → "回头客稳定" | ~40% |
| **L3 通用兜底** | 无规则命中时用 | "该产品数据表现良好，建议进一步沟通确认" | ~20% |

L3 虽然泛，但比"无数据"好。目标是 L1+L2 覆盖 80% 场景。

**依据：** [推测] Dario 判断 —— 规则引擎可控、零延迟、零成本。LLM 增强版排 V2。

---

### 问题 2：判词闪烁（阈值附近波动）

**原因：** 同一产品两次查询，回头率从 70.1% 变成 69.9% → 判词从"高回头率"变成"回头率一般" → 用户刷新看到判词变了 → 以为系统不稳定。

**主流方案：** 阈值加滞回区间（Hysteresis）：

```
触发阈值   > 70% → "高回头率"
回落阈值   < 65% → 才取消"高回头率"
```

65%-70% 之间的波动不触发判词变化。

**依据：** 经典控制理论，常见于温控、告警系统。防止"抖动"。

---

### 问题 3：新增品类需要改判词 → 改代码 → 发版

**原因：** 判词模板硬编码在 Python 里，新增品类（如服装、食品）需要开发改代码、测试、上线。

**主流方案：** 判词模板配置化 —— JSON 或 YAML 文件，放在 `src/api/prompts/` 或 `config/` 下。新增品类只需加配置，不动代码。运营或产品可以直接改。

```json
{
  "rules": [
    {
      "id": "high_repurchase",
      "level": "L1",
      "conditions": {"repeatRate": {">": 0.70}, "positiveReviewRate": {">": 0.95}, "tpYear": {">=": 5}},
      "verdict": "高信誉老店 · 回头率 {{repeatRate}}%",
      "tab": "factory"
    }
  ]
}
```

**依据：** 判词是业务逻辑，不是技术逻辑。配置与代码分离是基本原则。

---

### 问题 4：判词依赖的核心字段缺失

**原因：** Apify 返回数据不完整，判词规则的依赖字段（如 `repeatRate`）为 null → 规则引擎报错或判词为空。

**主流方案：** 判词引擎只依赖核心字段（价格、MOQ、回头率、好评率、tpYear、flags），并在计算前检查字段存在性。核心字段缺失 → 该维度判词降级为通用兜底；非核心字段缺失 → 不影响判词生成。

**依据：** 判词的底线是"不胡说"。数据不够时承认数据不够，比强行出判词好。

## 三、异常状态处理

### 问题 1：Apify 返回空数据

**原因：** 链接无效、商品已下架、1688 反爬升级、offerId 不存在。

**主流方案：** 三级降级响应：

| 级别 | 条件 | 响应 |
|------|------|------|
| **一级** | 有缓存（即使过期） | 返回缓存 + 顶部黄色警告条："数据可能不是最新，该链接当前无法获取" |
| **二级** | 无缓存 + 可识别原因（链接格式不对、offerId 不存在） | 返回明确错误 + 建议动作："该链接可能已下架，请检查后重试" |
| **三级** | 无缓存 + 原因不明（Apify 自身异常） | 返回通用错误 + 建议动作 + 人工客服入口："获取失败，请稍后重试。如持续失败请联系客服" |

**依据：** 张小龙 —— 异常路径占用户体验的 80%。正常路径好做，异常路径决定可靠性。

---

### 问题 2：Apify 返回部分数据（字段缺失）

**原因：** 1688 页面结构变化、某些产品没有某字段（如视频、SKU 图）、供应商信息不完整。

**主流方案：** 前端字段级容错渲染：

- 判词依赖的核心字段缺失 → 该维度判词用 L3 通用兜底
- 非核心字段缺失 → 静默隐藏该行（不显示空行，不报错）
- 所有字段渲染前做 null/undefined 检查

**禁止：** 因"视频 URL 为空"导致整个页面报错。

**依据：** 容错设计的基本原则 —— 能少显示不崩溃。

---

### 问题 3：用户输入非 1688 链接

**原因：** 用户贴了淘宝/拼多多/随便一段文字。如果穿透到 Apify → Apify 报错或返回空 → 浪费一次额度。

**主流方案：** 两级校验：

1. **前端**：正则校验 URL 格式 —— `1688.com` 或 `detail.1688.com` 才放行。不匹配 → 直接提示"请粘贴 1688 商品链接"，不发起请求
2. **后端**：收到请求后校验 offerId 是否存在且为数字。无效 → 返回 400，不调 Apify

**依据：** 防御性设计。不让无效请求消耗外部付费 API。

---

### 问题 4：后端自己挂了

**原因：** FastAPI 进程崩溃、内存溢出、依赖服务不可用。

**主流方案：**
- Health check endpoint (`/health`) 返回 200 + 基本状态
- Supervisor / systemd 配置 `restart=always`
- 日志写文件（`src/api/logs/` 已存在），方便排查
- V1 单机部署，不需要 K8s 或容器编排

**依据：** 单机最小运维。FastAPI 异步模式下资源占用低，核心风险是未捕获异常导致进程退出。Supervisor 自动拉起即可。

## 四、性能与加载体验

### 问题 1：用户等 20-40 秒什么也看不到

**原因：** Apify 调用是长耗时操作。如果后端同步阻塞等待 → HTTP 连接 hang 住 → 前端白屏 → 用户以为坏了就关了。

**主流方案：** 异步轮询模式（参考 OpenAI / Midjourney）：

```
前端                      后端                        Apify
 │                         │                           │
 │── POST /analyze ───────→│                           │
 │                          │── call actor ───────────→│
 │←─ {task_id, status:"pending"} ────────────────────  │
 │                          │                           │
 │── GET /task/{id} ───────→│ (每2s轮询)                │
 │←─ {status:"running", elapsed:18}                    │
 │                          │                           │
 │── GET /task/{id} ───────→│                           │
 │                          │←── 结果返回 ──────────────│
 │←─ {status:"done", result:{...}}                     │
```

等待期间：前端展示骨架屏（skeleton screen）+ 进度文案（"正在获取商品数据，预计还需 15 秒..."）。

**不推荐方案：** SSE / WebSocket。杀鸡用牛刀，增加运维复杂度（需要处理连接断开、重连、代理超时等）。

**依据：** 行业标准异步轮询模式。简单、可靠、可调试。

---

### 问题 2：轮询时用户切走页面再回来

**原因：** 手机锁屏、切微信回消息、网络切换。回来后前端状态丢失 → 轮询已停止 → 结果丢了 → 用户看到"网络错误"。

**主流方案：**
- `task_id` 存在 `sessionStorage`
- 页面加载时检查是否有未完成的 `task_id` → 有则恢复轮询
- 后端 task 结果保活 30 分钟（Redis 或内存字典 + TTL）

**依据：** 移动端常见场景。不做这个 → 用户等半天回来发现白等了 → 差评。

---

### 问题 3：首屏资源体积过大

**原因：** CSS/JS/字体过大 → 东南亚 4G 加载慢。

**主流方案：**
- Demo 的 CSS design token 系统直接复用（~2KB 变量定义）
- JS 按页面拆分：首页不加载 checkout/admin 逻辑
- 字体用系统栈（`-apple-system, BlinkMacSystemFont, 'Segoe UI'`），不引入 Web Font
- **目标：首屏 < 200KB（含 HTML+CSS+JS+图片占位）**

**依据：** 东南亚 4G 占比高。200KB 阈值参考 Google PageSpeed 建议。

## 五、判词多语言

### 问题 1：判词目前是中文，东南亚卖家看不懂

**原因：** Demo 面向中文测试，目标用户是越南/泰国/印尼卖家。

**主流方案：** V1 先出英文版判词，中文版保留。判词模板同时维护中英两个版本：

```json
{
  "id": "high_repurchase",
  "verdict_zh": "高信誉老店 · 回头率 {{rate}}%",
  "verdict_en": "Trusted Seller · {{rate}}% Repeat Buyers"
}
```

**不做：** 自动翻译 API（Google Translate / DeepL）——翻译质量不可控，翻译错了比没翻译更糟。

**依据：** [推测] 越南/泰国/印尼卖家英语读写能力参差不齐，但英文是最大公约数。V1.1 后按国家扩展本地语言版本。

---

### 问题 2：判词阈值要不要按国家调整？

**原因：** 越南卖家 vs 印尼卖家对"好价格""高利润"的判断不同。

**决定：V1 不调。** 判词只判断 1688 侧的客观指标（价格绝对值、回头率、工厂资质），不管目的地侧的利润判断。卖家自己会算账。

**依据：** 用户确认 ——"拿到低价他自己会算利润，别多此一举。"

## 六、用户行为边界情况

### 问题 1：用户连续快速点击"分析"按钮

**原因：** 焦虑——等了 3 秒没反应就猛点。如果不防重 → 多次请求 → 浪费 Apify 额度。

**主流方案：**
- **前端**：按钮点击后立即 `disabled = true` + 文案改为"分析中..."，`finally` 中恢复
- **后端**：同一 offerId 的未完成请求，直接返回已有 `task_id`（复用请求合并机制）

**依据：** CLAUDE.md 已有表单防重铁律。此处同理。

---

### 问题 2：用户在 WhatsApp 里分享 1688 链接给其他人

**原因：** 传播路径——但没有分享功能。

**决定：V1 不做分享功能。** 如果产品好用，用户自己会截图发。聚焦核心流程。

**依据：** 雷军说的传播——不靠功能，靠产品本身。

## 七、服务稳定性

### 问题 1：日志与监控缺失

**原因：** 出问题后不知道发生了什么。

**主流方案：**
- `src/api/logs/` 目录已存在，所有 Apify 调用记录到日志（成功/失败/耗时）
- 关键节点打日志：请求到达 → 缓存命中/未命中 → Apify 调用开始/结束/超时 → 响应返回
- V1 不需要接入监控平台（Prometheus/Grafana），tail 日志即可

**依据：** V1 用户量小，过度基建浪费时间。

---

### 问题 2：依赖 Apify 单点故障

## 八、数据库：查询历史存储

> 智囊团 2026-07-21 讨论结论。详见 §方案对比 方案 B。

### 核心决策：存，但只存元数据（方案 B）

**三种方案对比：**

| | 方案 A：不存 | 方案 B：存 offerId + 元数据 ✅ | 方案 C：存完整 JSON |
|------|------------|--------------------------|-------------------|
| **存什么** | 无 | offerId、标题、价格区间、图片URL、查询时间、用户ID | 方案 B + Apify 完整原始 JSON |
| **回看体验** | 无。每次重新查 | 可看历史列表，点进去走缓存 | 可离线查看历史（即使缓存过期） |
| **DB 行大小** | 0 | ~2KB/条 | ~50KB/条 |
| **数据新鲜度** | 总是最新 | 取决于缓存 | 可能过期 |
| **工期** | 0 天 | +0.5 天 | +1 天 |
| **运维** | 无 | 低 | 中（JSON 膨胀、需定期清理） |

**选方案 B 的理由：**

1. 1688 完整 JSON 存在 DB 没有意义——数据会过期，旧数据反而是误导
2. 存 offerId + 核心字段（标题、价格、图片），点进去时走缓存或重调 Apify——数据总是新鲜的
3. 单条记录 ~2KB，一万条才 20MB，MySQL 无压力
4. 有了 `query_history` 表，后续自然长出"热查排行"等增值功能

### 前端展示：极简"最近查询"

首页搜索框下方，最多 10 条。每条：缩略图 + 标题前 20 字 + 价格 + "X 小时前"。不翻页、不搜索、不筛选。

### 技术实现

```sql
-- db/schema.sql 新增
CREATE TABLE query_history (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    user_id VARCHAR(64) NOT NULL,
    offer_id VARCHAR(32) NOT NULL,
    title VARCHAR(200),
    price_min DECIMAL(10,2),
    price_max DECIMAL(10,2),
    image_url VARCHAR(500),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_user_time (user_id, created_at DESC)
);
```

**存储位置：** 后端 MySQL（不做前端 localStorage）——越南卖家反馈：换设备也能看。

### 明确不做（V1 范围外）

| 不做 | 原因 | 排期 |
|------|------|------|
| 收藏/标记 | 不是核心流程 | V2 |
| 产品对比 | 需要数据积累和独立 UI | V2 |
| 历史分页/搜索 | 10 条以内不需要 | 不做 |
| 完整 JSON 存储 | 数据过期没意义 | 不做 |

## 九、AI 决策：V1 零 AI

> 智囊团 2026-07-21 讨论结论。Dario 主笔。

### 核心决策：V1 不使用任何 AI

**AI 可能的三处应用及 V1 方案：**

| 位置 | AI 能做什么 | V1 方案 | V2 演进 |
|------|-----------|---------|---------|
| **① 判词生成** | LLM 根据完整数据写总结判词 | 规则引擎（L1+L2+L3 三层） | LLM 增强版（前提：规则引擎跑一年 + 数据积累） |
| **② 产品翻译** | 中文标题/规格 → 英文/越南语/泰语 | 判词模板中英双语手动维护，不做自动翻译 | DeepL API / LLM 翻译 |
| **③ 智能推荐** | "同类低价替代品" | 不碰（无数据基础） | 需 ≥1000 条查询记录 + 用户行为 |

### 为什么不用

**功能层面：**

1. **规则引擎覆盖 80% 场景**，零成本、零延迟、不幻觉。LLM 判词有幻觉风险——说错一句话（"该产品已通过 ISO 认证"），卖家基于它下单，责任谁担？
2. **翻译质量不可控**。Google Translate / DeepL 翻译专业术语（"不锈钢"、"冷轧板"）可能出错，翻译错了比没翻译更糟
3. **推荐需要数据基础**。V1 日活 < 10 人，没有足够数据做任何形式的推荐

**营销层面（雷军）：**

- V1 **不标榜 AI**。如果功能本身没有 AI，不要硬贴 AI 标签。卖家不傻，说了 AI 体验不到 → 反而觉得被骗
- V2 接了 LLM 后，不需要"贴标签"——用户自己会用出来。那时的营销话术是"已被 3000+ 东南亚卖家使用"，比"AI 驱动"管用 10 倍

**产品层面（张小龙）：**

- 用户不关心你用了什么技术。他关心的是：**贴链接 → 出结果 → 帮我做决策**
- 判词准不准、页面快不快、数据新不新——这三样 V1 都可以不用 AI 做到

### V2 引入 AI 的前置条件

| 条件 | 说明 |
|------|------|
| 规则引擎跑 ≥ 6 个月 | 积累"哪些判词用户觉得有用"的数据 |
| 查询历史 ≥ 5000 条 | 够做基础推荐和热度排序 |
| LLM 判词标注"AI 生成，仅供参考" | 降低幻觉风险的法律责任 |
| 翻译准确率 ≥ 95% 验收标准 | 产品术语白名单 + 人工抽检 |

### 产品定位

**V1 = 数据展示 + 规则引擎产品**，不是 AI 产品。V2 才是 AI 产品。
