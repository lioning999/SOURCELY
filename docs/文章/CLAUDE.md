## AI 铁律（不遵守就会出问题）

### 一、手术式修改

**只动必须改的。只清理自己的烂摊子。**
编辑已有代码时：
- 不要"改进"相邻代码、注释或格式
- 不要重构没坏的东西
- 匹配现有风格，即使你有不同做法
- 如果注意到无关的死代码，指出它——不要删
当改动产生孤立代码时：
- 删除因你的改动而不再使用的 import/变量/函数
- 不要删除已存在的死代码，除非被要求
**检验标准：每一行改动都应直接追溯到用户的需求。**

### 二、先读后写（铁律）
**改文件前先读文件。新增功能前先读同层现有结构。**
- 改任何文件前，先读它的当前内容——不凭记忆或推测写
- 新增功能前，先读同层现有文件的结构和命名
- 新增 API 端点前，先 grep 已有路由确认没有重复
- 如果 AI 之前改过这个文件但不在当前上下文中 → 先读当前内容再改
**禁止：凭训练数据里的"典型写法"直接写代码。所有修改基于当前项目文件内容。**

### 三、简单至上（铁律）
**最少代码解决问题。不写推测性代码。**
- 不添加需求之外的功能
- 不为一次性代码创建抽象层
- 不添加未被要求的"灵活性"或"可配置性"
- 不为不可能发生的场景添加错误处理
- 如果写了 200 行但 50 行能搞定，重写
**自问："该项目里的任何人看了会觉得过度设计吗？" 如果是，简化。**

### 四、修改联动检查（铁律）

**改一个变量/常量/组件前，grep 全项目所有引用点。**
同一个概念（数值、颜色、间距、逻辑模式）出现在 ≥3 个文件中 → **必须先抽取为共享定义**，再改。
- 改 CSS 固定尺寸 → grep 该值的所有 `.css` 引用
- 改 JS 工具函数 → grep 所有调用点，确认每个调用者的上下文
- 改 Python 常量 → grep 所有 import 和使用点
- 改 API 路由 → grep 前端所有调用该路由的代码
- 改 API 响应格式 → 确认前端对应解析逻辑同步修改（不改前端的响应字段=前端炸）
- 改函数签名 → grep 所有调用方，确认参数对齐
**违反后果：改 A 漏 B，修一个 bug 引出另一个。**

### 五、异步竞态防御（铁律）
任何 `setInterval` / `setTimeout` / `asyncio.create_task` / 全局可变状态，必须遵循：
1. **创建新的之前必须先清理旧的**——入口第一行执行 `clearInterval` / `cancel` / `cleanup`
2. **达到终止条件时，先杀自己再通知外部**
3. **全局可变容器（dict/list/queue）必须有上限和清理机制**

### 六、修改边界（铁律）
| 类型 | 能否修改 |
|------|---------|
| `CLAUDE.md` | ❌ 需要用户明确授权 |
| `docs/项目架构宪法.md` | ❌ 需要用户明确授权 |
| 项目源代码 | ✅ |
| `.claude/` / `.vscode/` 下的所有文件 | ❌ |
| 系统级/用户级配置 | ❌ |
**讨论不算授权——用户说"执行""改"才算。**

### 七、违宪自检 + 三级违规处理（铁律）
**每次代码提交前，必须自检是否违反项目架构宪法。**
检查清单：
1. 检查改动是否越层调用（如：Controller 直接写 SQL）
2. 检查改动是否引入未授权的技术栈依赖
3. 检查改动是否绕过安全规则（如：未校验用户身份）
4. 检查改动是否改了接口格式没通知前端
5. 检查代码报错是否主动暴露给了用户（禁止默默修复不告知）
发现违宪后，按三级处理：
| 级别 | 判定 | 处理方式 |
|------|------|---------|
| **一级**（架构破坏） | 越层调用、绕过安全中间件、引入黑名单技术栈 | PR 直接拒绝，修复后才能提交 |
| **二级**（关键缺失） | 未校验用户身份、死代码未清理、新建文件未注册 | 24h 内修复，修复前阻塞合并 |
| **三级**（规范不符） | 函数超 50 行、命名不一致、注释写 WHAT | 提醒修正，不阻塞 |
**不隐藏错误：** 运行报错时，主动报告错误信息和上下文，不要偷偷重试或绕过去。

### 八、多步任务先列计划（铁律）
**超过 3 步的任务，先列计划再执行。**
格式：
```
1. [步骤] → 验证: [检查项]
2. [步骤] → 验证: [检查项]
3. [步骤] → 验证: [检查项]
```
- 计划列出后等用户确认再开始执行
- 不列计划直接开写 → 写偏了重来成本更高
- 如果用户说"直接做" → 可以跳过，但完成后补一个"做了什么"的总结

## 代码约定

### 命名一致性
- 文件名、函数名、变量名保持统一模式
- 函数名动词+名词（create_order、get_member_info）
- 文件名跨层对齐：`routes/auth.py` ↔ `services/auth.py`

### 类型与注释
- 所有函数必须有完整类型注解（参数 + 返回值）
- 注释写 WHY，不写 WHAT——WHAT 从命名和类型就能读出来
- 每个文件顶部一行描述用途

### 死代码清理
**删除任何函数/类/变量前，必须 grep 全项目所有引用点。** 确认零引用后才能删除。

### 新建文件约束
**不新建文件优先。扩展现有文件优于新建文件。**
确实需要新建时：
1. grep 确认没有同功能已有文件
2. 确认新文件注册到入口（路由注册 / 模块引入 / 入口引用）
3. 文件名必须符合项目命名规范
4. 不要新建"公共工具类"文件——优先放到已有同类文件中
5. 新建文件后，在文件顶部一行描述其用途

## 输出格式
- 结论先行，再给理由
- 被指出错误时：承认 → 修正，不辩解
- 不确定时：标注置信度——确定 / 推测（建议验证） / 存疑（不可做决策依据）

## 文档创建规则
1. 功能不简单（非单接口 CRUD）→ 创建 `docs/需求-{主题}.md`
2. 设计方案无法从代码推导 → 创建 `docs/设计-{主题}.md`
3. 创建新文档加目录索引，更新 `docs/README.md` 索引
4. 禁止创建空文档

## 首次使用：初始化项目

当用户说 **"初始化项目"** 时，执行以下流程：

### 步骤

1. **读配置** → 读取 `README.md` 末尾的「初始化配置」段，提取 6 个值
2. **生成/更新文件** → 按配置修改以下文件：

   | 文件 | 改动 |
   |------|------|
   | `docs/项目架构宪法.md` | 填充 `{{TODO}}` 占位符 |
   | `requirements.txt` | 按数据库选型启用对应驱动 |
   | `main.py` / `middleware.py` | 按认证方式启用/关闭中间件 |
   | `src/api/CLAUDE.md` | 如有需要调整数据库示例 |

3. **输出修改清单** → 列出改了哪几个文件、具体改了什么

### 配置映射规则

```
技术栈=Python+FastAPI → 骨架代码不动
数据库=MySQL          → requirements.txt 启用 aiomysql
数据库=PostgreSQL     → requirements.txt 启用 asyncpg
数据库=SQLite         → requirements.txt 启用 aiosqlite
认证方式=JWT           → middleware.py 启用 JWT 中间件
认证方式=无需认证      → middleware.py 注释掉 JWT 中间件
部署方式=Docker       → 项目架构宪法 §部署填充 Docker 模板
```

### 注意事项

- 只改上述文件，**不要改其他代码**
- 配置值有歧义时向用户确认
- 改完后输出修改清单，**必须等用户确认再执行后续动作**
