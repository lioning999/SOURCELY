## 开头（0:00–0:30）

你有没有发现——AI写代码的时候看起来很聪明，一运行全是bug？

更诡异的是：它不是不会写。它是写出来了，但写的不是你想要的。

我用了两年AI编程，市面上主流的AI编程工具和模型基本都用过，踩了上百个坑。最后发现一件事：

**问题不全在AI，在你跟它说话的方式和给它的边界约束上。**

今天这期，我讲5个最容易翻车的问题。每个都告诉你为什么错、怎么改。改完之后，AI代码准确率一定会翻倍提升。

**第一个问题最致命——它让你觉得代码能用，实际上全是坑。**


## 五阶段框架（0:30–0:55）

这五个问题不是随便列的。它们对应你跟AI协作的五个阶段，一个套一个：

**第一阶段：需求说不清。** 你脑子里有场景画面，AI不知道——它只能猜。
**第二阶段：任务拆不开。** 一次给太多，AI前面记得后面忘。
**第三阶段：结构没设计。** 功能多了，AI全堆在一个文件里，你改不动。
**第四阶段：修改没规范。** 改A崩B，修B坏C——连锁反应。
**第五阶段：结果没验证。** 代码能跑，你就以为没事了。其实漏洞全藏在里面。

这五步是个死循环。你把每一环补上，AI就会从"乱写代码的实习生"变成能配合你的开发助手。

一个一个拆。


## 问题一：需求说偏了（0:55–2:30）

你说"帮我做一个登录"。

你脑子里想的是：手机号输入框、获取验证码按钮、六位数字输入、验证成功跳首页。

AI理解的是：用户名输入框、密码输入框、登录按钮。

它没做错——它只是不知道你脑子里还有"手机号""验证码""短信""跳首页"这些信息。你没告诉它的，它只能猜。

**把AI当成刚入职第一天的实习生。** 实习生不知道你的代码习惯、项目结构、甚至你说的"登录"是哪种登录。嚼碎了喂。

怎么做？用一个标准公式。每次跟AI说话，按这个填：

```
我要做：【具体功能】

使用场景：【谁、什么情况下用】

交互流程：
1. 用户先……
2. 然后……
3. 如果成功→……
4. 如果失败→……

技术约束：
- 框架：【 】
- 文件放在：【 】

限制：
- 已有功能不要动：【 】

开始前请你：
1. 复述我的需求，确认你理解了
2. 列出要改哪些文件、每个文件负责什么
3. 等我确认再写代码
```

顺便说一下——我说的"准确率翻倍"不是AI永远不会出错。是返工次数减半：原来10次有8次要大改，改完大概3到4次。不是玄学，是体验。

好，第二个问题。


## 问题二：一次塞太多（2:30–3:40）

你一定干过这种事：

"帮我做个商城——注册、登录、商品列表、详情、购物车、支付、会员、积分、订单、退款、收藏、评价、优惠券……哦对了，全部用绿色！"

AI写到购物车的时候——绿色？什么绿色？已经给你用上蓝色了。

AI有一个叫上下文窗口的东西，你可以理解为一个杯子。你往里扔的东西越多，前面的就被挤出去了。十几个功能一口气扔进去，写到第五个，前四个的规则早被冲出杯子了。

**每次只做一件事。做完、跑通、确认没问题——再开下一个。**


## 问题三：结构没设计（3:40–5:10）

这是前两个问题的"并发症"——需求没说清、一次又塞太多，AI把所有功能全写进一个文件。

我见过最离谱的：App.vue，1700行。登录、商品列表、个人中心、支付回调、订单管理，五个模块全在里面。想改支付按钮颜色——翻到600行找到按钮，改了颜色，不小心删了旁边一个函数。后来发现那是订单查询的核心逻辑。一个颜色改了3个小时。

为什么会这样？AI默认你不在乎结构。你说"帮我做xxx"，它选最快的方式——全写当前文件里。

**让AI写代码之前，先列文件结构。**

```
src/
├── modules/
│   ├── auth/          ← 登录、注册
│   ├── product/       ← 商品列表、详情
│   ├── order/         ← 下单、支付
│   └── user/          ← 个人信息
├── components/        ← 可复用组件
├── services/          ← API请求
└── utils/             ← 工具函数
```

先画地图，再走路。


## 问题四：改A坏B（5:10–6:30）

这个问题最让人崩溃——你调了三天的功能，改一个按钮颜色，另一个功能炸了。

我就干过：登录按钮改成红色→支付按钮消失。找回支付按钮→订单提示没了。修好订单提示→页面样式全乱。你连哪个版本能用都分不清。

为什么会这样？AI每次只盯着你指的那一个点改。它不知道你的全局规则——按钮颜色规范、命名方式、哪些文件不能碰。

**给你的项目建一个规矩文件。** 把所有规范写进去。以后每次让AI改代码之前，先贴给它：

```markdown
# 项目规范 — AI修改代码前必读

## 技术栈
- 前端：【 】
- 后端：【 】
- 数据库：【 】

## 文件结构
- 页面放：【 】
- 组件放：【 】
- API请求放：【 】

## 修改铁律
- 不要删除任何已有功能
- 修改前先列出受影响的所有文件
- 每次只改一个功能点
- 改完后列出测试步骤

## UI规范
- 主色调：【 】
- 主按钮样式：【 】
- 危险按钮样式：【 】

## 禁止事项
- 不要用 any 类型
- 不要在页面里直接写请求
- 不要留 console.log
```

这个规矩文件，放评论区了。直接复制就能用。


## 问题五：能跑 ≠ 没问题（6:30–7:50）

最后一个问题最隐蔽——它不报错。

我上周刚经历了一次：让AI写一个用户信息收集的表单。写完一跑——能提交、没报错、页面正常。差点就过了。

想起来要多问一句，敲了三个字："安全吗？"

AI的回答让我后背一凉：接口没有鉴权——任何人知道URL就能调。SQL没有参数化——可以被注入。参数没有校验——能通过改请求参数拿到别人的数据。

三个问题，全在暗处。我让它当场修复，多花了5分钟。

隔壁团队没做这一步。两周后，他们的用户手机号被人批量爬走了。

**"能跑"和"能上线"之间，隔了一整套防护。** 养成习惯——AI写完代码，问三句话：

```
1. "这段代码有没有安全隐患？列出风险点，按严重程度排。有就直接修复。"

2. "有没有性能问题？重复请求、N+1查询、没做缓存、内存泄漏？"

3. "帮我写3个测试用例，覆盖正常流程和异常流程。我自己验证一遍。"
```

让AI自己查一遍，你再验一遍。三句话贴编辑器旁边。


## 结尾（7:50–8:40）

好，五个问题全部拆完了。

如果你现在正在用AI写代码，下一次不要上来就说"帮我做个功能"。**先让AI做三件事：**

第一，复述你的需求——确认你俩说的是同一件事；
第二，列出文件结构——确认代码放对地方；
第三，给出测试步骤——确认你知道怎么验证结果。

这三步做了，踩坑概率少一半以上。
