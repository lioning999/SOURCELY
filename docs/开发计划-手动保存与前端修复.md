### 改 2：新增 `POST /api/save-report`

| 项 | 内容 |
|----|------|
| **文件** | `src/api/routes/analyze.py`（新增路由，不动现有代码） |
| **逻辑** | `POST /api/save-report {offer_id}` → JWT 取 user_id → 从内存缓存取 mapped 数据 → 调 `_save_to_db()` → 返回 |
| **复用** | `analyze_service` 已有完整的 `_save_to_db` 链路（18 字段 + 3 子表 + 事务） |

```
POST /api/save-report
  1. JWT 认证 → user_id（必登录，不登录直接 401）
  2. body: { offer_id: "1047302552732" }
  3. 从 domain/cache.py 内存缓存读 mapped 数据
  4. 缓存未命中 → 返回 410 "分析已过期，请重新搜索"
  5. 缓存命中 → await analyze_service._save_to_db(mapped, user_id, offer_id)
  6. 返回 { code: 200, data: { saved: true }, message: "已保存" }
```

| **注意事项** | |
|--------------|-----|
| ⚠️ 重复保存 | `uk_offer_user (offer_id, user_id)` 唯一键。第二次保存同一商品会爆 `IntegrityError` |
| 处理方式 | 用 `INSERT ... ON DUPLICATE KEY UPDATE` 或先查后决。**推荐 ON DUPLICATE KEY UPDATE**——用户可能想更新（商品降价了），覆盖旧记录合理 |
| ⚠️ 缓存过期 | TTL 30min，超过 30min 用户再点保存会失败。返回明确告知"分析已过期，请重新搜索" |
| ⚠️ save_to_db 是 async | route 里直接 await 它，不走 asyncio.create_task（用户点按钮要即时反馈） |
| **安全** | offer_id 必须校验格式（纯数字，长度 ≥ 8），防参数注入 |

### 改 3：前端保存按钮

| 项 | 内容 |
|----|------|
| **文件** | `src/web/report.html` + `src/web/static/js/pages/report.js` |
| **触发时机** | 分析完成后（`pollTask` 收到 `status=done`），在卡片区域上方显示保存按钮 |
| **状态机** | |

```
[未登录] → 不显示保存按钮（或显示灰的"登录后保存"）
[已登录·未保存] → 显示蓝色"保存到我的分析"
[点击中] → 按钮 disabled + "保存中..."
[已保存] → 按钮变灰 + "✓ 已保存"
[缓存过期] → 按钮变红 + "已过期，请重新搜索"
```

| **注意事项** | |
|--------------|-----|
| ⚠️ 防重 | 点击后立即 disabled，finally 不启用。状态持久化：`sessionStorage.setItem('saved_' + offerId, '1')`，刷新页面后仍显示"已保存" |
| ⚠️ 跨页面 | 从历史页点进来 → URL 带 offerId → 自动分析 → 如果之前保存过，检查 sessionStorage 直接显示"已保存" |
| ⚠️ 按钮位置 | 放在卡片顶部（判词下面），或 WhatsApp 按钮旁边。**不要藏在拿样 Tab 里** |

### 改 4：工厂 Tab 选择器修正

| 项 | 内容 |
|----|------|
| **文件** | `src/web/static/js/pages/report.js`（`renderResult` 函数内 3 行） |
| **根因** | |

```
report.html DOM：
#tab-factory
  ├─ :nth-child(1) .verdict
  ├─ :nth-child(2) .sec "基本信息"      ← 不是 .row！
  ├─ :nth-child(3) .row 公司名
  ├─ :nth-child(4) .row 地址
  └─ ...

report.js 写的是：
  .row:nth-child(2) → 匹配到 .sec，不是 .row → null
  .row:nth-child(3) → 匹配到地址（碰巧对了 .row，但跳过了公司名）
```

**修复方式：** `querySelectorAll` 按索引取：

```javascript
// 旧（错）
var nameRow = document.querySelector('#tab-factory .row:nth-child(2) .row-value');
var addrRow = document.querySelector('#tab-factory .row:nth-child(3) .row-value');

// 新（对）
var rows = document.querySelectorAll('#tab-factory .row');
// rows[0] → 公司名, rows[1] → 地址, rows[2] → 商家身份, 
// rows[3] → 工厂实力, rows[4] → 认证, rows[5] → 排名
var nameRow = rows[0]?.querySelector('.row-value');
var addrRow = rows[1]?.querySelector('.row-value');
```

| **注意事项** | |
|--------------|-----|
| ⚠️ 只改公司名和地址两处 | renderResult 里确实只用了这两个字段。店铺等级、认证、排名的 DOM 不动（保持 demo 显示） |
| ⚠️ 兼容 | `querySelectorAll` 所有浏览器都支持，无兼容性风险 |

## 五、风险矩阵

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| DB ALTER 锁表 | 低 | 中 | analysis 表现无数据，秒级完成 |
| 缓存过期用户点保存 | 中 | 低 | 返回明确提示"重新搜索"，用户可接受 |
| 用户快速双击保存 | 中 | 低 | 前端 disabled + 唯一键兜底 |
| ON DUPLICATE KEY UPDATE 覆盖旧数据 | 低 | 低 | 商品信息更新是预期行为（价格变动） |
| report.js 选择器改错 | 低 | 中 | 只改公司名+地址两处，改完测试验证 |
