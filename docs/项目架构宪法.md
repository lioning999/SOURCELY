## 二、系统分层架构

### 2.1 分层总览

```
请求 → CORS → JWT中间件 → routes/（参数校验）
       → services/（业务编排）
         → adapters/（外部API）
         → repositories/（数据库）
         → domain/（纯逻辑）
```

### 2.2 各层职责

| 层 | 职责 | 依赖 |
|----|------|------|
| **routes/** | Pydantic 参数校验 + 调用 service + 标准返回 | 只依赖 services/ |
| **services/** | 业务编排、事务边界、注入 adapter/repo | 依赖 adapters/ + repositories/ + domain/ |
| **adapters/** | 外部 API 封装（Apify / Google OAuth，一个服务一个文件） | 不依赖其他层 |
| **repositories/** | 数据库操作（连接生命周期自管） | 不依赖其他层 |
| **domain/** | 纯函数、零外部依赖 | 不依赖任何层 |
| **utils/** | 通用工具（异常/JWT/日志/配置） | 不依赖 services/ 或 routes/ |

### 2.3 层间通信规则（铁律）

1. **上层依赖下层，下层不依赖上层** — routes → services → repositories/adapters/domain
2. **同层不互调** — `service_a.py` 不能调 `service_b.py`，公共逻辑提取到 domain/
3. **越层调用 = 违宪** — routes 写 SQL、services 手管连接，发现 PR 直接拒绝

## 三、代码标准形状

### 3.1 Repository 标准

> 一条 SQL 对应一个方法，一个表对应一个 Repository 文件。

```python
# repositories/analysis_repo.py — 分析记录表操作
class AnalysisRepository:
    """数据库操作封装。事务由 service 层控制。"""

    def __init__(self, db: DatabaseConnection):
        self.db = db

    async def get_by_offer_id(self, offer_id: str) -> dict | None:
        async with self.db.get_connection() as conn:
            async with conn.cursor() as cur:
                await cur.execute(
                    "SELECT id, offer_id, status, price_min, title FROM analysis WHERE offer_id=%s",
                    (offer_id,)
                )
                return await cur.fetchone()

    async def create(self, data: dict) -> int:
        async with self.db.get_connection() as conn:
            async with conn.cursor() as cur:
                await cur.execute(
                    "INSERT INTO analysis (offer_id, status, title) VALUES (%s, %s, %s)",
                    (data["offer_id"], data["status"], data["title"]),
                )
                await conn.commit()
                return cur.lastrowid
```

**规则：**
- 方法名以 `get_` / `create_` / `update_` / `delete_` / `count_` / `exists_` 开头
- 返回 `dict` 或 `None`，不返回 ORM 对象
- 一个文件一个表，文件名与表名对齐

### 3.2 Service 标准

> 编排业务逻辑，不写 SQL，不管连接。

```python
# services/analysis_svc.py
class AnalysisService:
    def __init__(self, repo: AnalysisRepository, apify: ApifyAdapter):
        self.repo = repo
        self.apify = apify

    async def analyze(self, offer_id: str, user_id: int) -> dict:
        cached = await self.repo.get_by_offer_id(offer_id)
        if cached and cached["status"] == "done":
            return cached
        result = await self.apify.fetch_product(offer_id)
        verdict = await self.engine.judge(result)
        await self.repo.create({**result, **verdict, "user_id": user_id})
        return {**result, **verdict}
```

**规则：**
- 方法名用业务动作命名：`register()` / `submit_order()` / `cancel_subscription()`
- 事务边界在 service 方法内，不在 route 层
- service **只抛 AppError 子类**，不 raise `HTTPException` 或 `ValueError`

### 3.3 Adapter 标准

> 外部 API 调用封装。一个外部服务一个文件。

```python
# adapters/apify_adapter.py
class ApifyAdapter:
    """Apify 1688 商品抓取适配器。使用 ApifyClientAsync SDK。"""

    def __init__(self, token: str = ""):
        self.token = token or Config.APIFY_TOKEN

    async def fetch_product_by_url(self, url: str) -> dict[str, Any] | None:
        """actor.call() 启动 run → 等待完成 → dataset.list_items() 取结果。"""
        try:
            client = ApifyClientAsync(token=self.token)
            run = await client.actor(Config.APIFY_ACTOR_ID).call(
                run_input={"offerIds": [offer_id]},
                wait_duration=timedelta(seconds=Config.APIFY_WAIT_SECONDS),
            )
            if run is None:
                return None
            page = await client.dataset(run.default_dataset_id).list_items(limit=1)
            items = page.items
            return items[0] if items else None
        except ApifyApiError as e:
            raise ExternalServiceError(service_name="Apify", details={"offer_id": offer_id}) from e
```

**规则：**
- 所有外部异常包装为 `ExternalServiceError`
- 超时策略在 adapter 内实现（`wait_duration`）
- adapter 不包含业务逻辑
- ApifyClientAsync v3.x 不支持 `async with`，直接实例化

## 四、数据架构

### 4.1 数据库选型

- **引擎：** MySQL 8.0 / InnoDB / utf8mb4
- **驱动：** aiomysql 异步连接池（5-20）
- **连接池设 `autocommit=False`**，事务由 Repository 层显式控制

### 4.2 表结构管理

- **表结构以 `db/schema.sql` 为准**
- 当前表：`users` / `analysis` / `analysis_spec` / `analysis_sku` / `analysis_price_tier`
- 新增/修改字段必须先改 `schema.sql`，再改 Repository
- `schema.sql` 纳入版本管理，合并前人工 review

### 4.3 命名规范

| 对象 | 规范 | 示例 |
|------|------|------|
| 数据库 | 小写英文 | `my_project` |
| 表 | 复数小写 | `users`、`order_items` |
| 字段 | 小写下划线 | `created_at`、`order_status` |
| 主键 | `id` BIGINT AUTO_INCREMENT | — |
| 外键 | `{表}_id` | `user_id`、`order_id` |
| 时间戳 | `created_at` / `updated_at` | — |
| 软删除 | `deleted_at` DATETIME NULL | — |

### 4.4 SQL 铁律

- **禁止字符串拼接 SQL** — 100% 参数化 `%s`
- **动态字段名用白名单校验** — 禁止直接拼接列名
- **复杂查询写视图或存储过程**，不在代码中拼动态 SQL
- **禁止 SELECT \*** — 必须显式列出字段

## 五、安全架构

### 5.1 认证

- **认证方式：** Google OAuth 2.0 + JWT（HS256，7 天过期）
- **流程（服务端 OAuth 2.0）：**
  1. `GET /api/auth/google/login` → 302 重定向到 Google 授权页
  2. 用户授权 → Google 回调 `GET /google-callback?code=xxx`
  3. 后端 code 换 id_token（调 Google token 端点）
  4. 验证 id_token（aud / exp / iss）→ Upsert users 表 → 签发自签 JWT
  5. 302 重定向首页，token 通过 URL query 传递
- Token 存储：请求头 `Authorization: Bearer <token>`
- 密钥来源：环境变量 → `Config`，禁止硬编码
- 详见 `docs/设计-Google登录方案.md`

### 5.2 授权

- 路由分三级：**公开 / 半公开（未登录走配额）/ 需登录**
- 公开路径：`GET /api/analyze/{task_id}`（分析结果）、`GET /api/auth/google/login`（登录入口）、`GET /google-callback`（OAuth 回调）
- 半公开路径：`POST /api/analyze`（启动分析）— 未登录用户走 IP 每日配额（3 次），登录用户无限制
- 需登录路径：`GET /api/history`（历史记录）
- 权限校验在 JWT 中间件完成
- 用户身份统一从 `request.state.user_id` 获取，禁止前端传入

### 5.3 数据安全

- 无密码存储（Google OAuth，不存密码）
- 敏感字段：email 在日志中脱敏（前 3 位 + `***@***`）
- 密钥/Token 禁止进日志、禁止进响应体
- 所有用户输入在 route 层用 Pydantic 校验

## 六、API 设计规范

### 6.1 风格

- **RESTful**：`POST /api/analyze`（启动分析）、`GET /api/analyze/{task_id}`（轮询结果）
- URL 路径：小写字母 + 连字符
- 不使用 `/api/v1/` 前缀（V1 无版本管理需求）

### 6.2 响应格式

标准统一：`{code: 200, data: ..., message: "ok"}`
例外接口（不包装）：`GET /google-callback` 返回 `RedirectResponse`（302 重定向首页带 token）

### 6.3 错误码

| 范围 | 含义 |
|------|------|
| 400 | 参数校验失败 / Google token 无效 |
| 401 | JWT 过期 / 未登录 |
| 404 | 分析记录不存在 |
| 500 | 服务器内部错误 |
| 502 | Apify 外部服务异常 |
