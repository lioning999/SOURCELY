## 二、系统分层架构

### 2.1 分层总览

```
请求 → CORS → JWT中间件 → routes/（参数校验）
       → services/（业务编排）
         → adapters/（外部API）
         → repositories/（数据库）
         → domain/（纯逻辑）
```

### 2.2 各层职责

| 层 | 职责 | 依赖 |
|----|------|------|
| **routes/** | Pydantic 参数校验 + 调用 service + 标准返回 | 只依赖 services/ |
| **services/** | 业务编排、事务边界、注入 adapter/repo | 依赖 adapters/ + repositories/ + domain/ |
| **adapters/** | 外部 API 封装（一个外部服务一个文件） | 不依赖其他层 |
| **repositories/** | 数据库操作（连接生命周期自管） | 不依赖其他层 |
| **domain/** | 纯函数、零外部依赖 | 不依赖任何层 |
| **utils/** | 通用工具（异常/JWT/日志/配置） | 不依赖 services/ 或 routes/ |

### 2.3 层间通信规则（铁律）

1. **上层依赖下层，下层不依赖上层** — routes → services → repositories/adapters/domain
2. **同层不互调** — `service_a.py` 不能调 `service_b.py`，公共逻辑提取到 domain/
3. **越层调用 = 违宪** — routes 写 SQL、services 手管连接，发现 PR 直接拒绝
4. **数据跨层用 Pydantic 模型**，禁止传 dict

## 三、代码标准形状

### 3.1 Repository 标准

> 一条 SQL 对应一个方法，一个表对应一个 Repository 文件。

```python
# repositories/users.py — 用户表操作
class UserRepository:
    """数据库操作封装。事务由 service 层控制。"""

    def __init__(self, db: DatabaseConnection):
        self.db = db

    async def get_by_id(self, user_id: str) -> dict | None:
        async with self.db.get_connection() as conn:
            async with conn.cursor() as cur:
                await cur.execute("SELECT * FROM users WHERE id=%s", (user_id,))
                return await cur.fetchone()

    async def create(self, data: dict) -> str:
        async with self.db.get_connection() as conn:
            async with conn.cursor() as cur:
                await cur.execute(
                    "INSERT INTO users (name, email) VALUES (%s, %s)",
                    (data["name"], data["email"]),
                )
                await conn.commit()
                return cur.lastrowid
```

**规则：**
- 方法名以 `get_` / `create_` / `update_` / `delete_` / `count_` / `exists_` 开头
- 返回 `dict` 或 `None`，不返回 ORM 对象
- 一个文件一个表，文件名与表名对齐

### 3.2 Service 标准

> 编排业务逻辑，不写 SQL，不管连接。

```python
# services/user_service.py
class UserService:
    def __init__(self, repo: UserRepository, email_adapter: EmailAdapter):
        self.repo = repo
        self.email = email_adapter

    async def register(self, data: dict) -> dict:
        if await self.repo.exists_by_email(data["email"]):
            raise ValidationError("邮箱已被注册")
        user_id = await self.repo.create(data)
        # 异步通知，不阻塞注册流程
        asyncio.create_task(self.email.send_welcome(data["email"]))
        return await self.repo.get_by_id(user_id)
```

**规则：**
- 方法名用业务动作命名：`register()` / `submit_order()` / `cancel_subscription()`
- 事务边界在 service 方法内，不在 route 层
- service **只抛 AppError 子类**，不 raise `HTTPException` 或 `ValueError`

### 3.3 Adapter 标准

> 外部 API 调用封装。一个外部服务一个文件。

```python
# adapters/email_adapter.py
class EmailAdapter:
    """邮件发送适配器。封装具体调用细节。"""

    def __init__(self, api_key: str, base_url: str):
        self.api_key = api_key
        self.base_url = base_url

    async def send(self, to: str, subject: str, body: str) -> bool:
        try:
            async with httpx.AsyncClient() as client:
                resp = await client.post(
                    f"{self.base_url}/send",
                    json={"to": to, "subject": subject, "body": body},
                    headers={"Authorization": f"Bearer {self.api_key}"},
                    timeout=10,
                )
                resp.raise_for_status()
                return True
        except (httpx.TimeoutException, httpx.HTTPStatusError) as e:
            raise ExternalServiceError(service_name="email", detail=str(e)) from e
```

**规则：**
- 所有外部异常包装为 `ExternalServiceError`
- 超时和重试策略在 adapter 内实现
- adapter 不包含业务逻辑

## 四、数据架构

### 4.1 数据库选型

- **引擎：** {{MySQL 8.0}} / InnoDB / utf8mb4
- **驱动：** {{aiomysql}} 异步连接池（5-20）
- **连接池设 `autocommit=False`**，事务由 Repository 层显式控制

### 4.2 表结构管理

- **表结构以 `db/schema.sql` 为准**，截至时间 {{YYYY-MM-DD}}
- 新增/修改字段必须先改 `schema.sql`，再改 Repository
- `schema.sql` 纳入版本管理，合并前人工 review

### 4.3 命名规范

| 对象 | 规范 | 示例 |
|------|------|------|
| 数据库 | 小写英文 | `my_project` |
| 表 | 复数小写 | `users`、`order_items` |
| 字段 | 小写下划线 | `created_at`、`order_status` |
| 主键 | `id` BIGINT AUTO_INCREMENT | — |
| 外键 | `{表}_id` | `user_id`、`order_id` |
| 时间戳 | `created_at` / `updated_at` | — |
| 软删除 | `deleted_at` DATETIME NULL | — |

### 4.4 SQL 铁律

- **禁止字符串拼接 SQL** — 100% 参数化 `%s`
- **动态字段名用白名单校验** — 禁止直接拼接列名
- **复杂查询写视图或存储过程**，不在代码中拼动态 SQL
- **禁止 SELECT \*** — 必须显式列出字段

## 五、安全架构

### 5.1 认证

- **认证方式：** {{JWT（access_token + refresh_token）}}
- Token 存储：请求头 `Authorization: Bearer <token>`
- 密钥来源：环境变量 → `Config`，禁止硬编码

### 5.2 授权

- 路由分三级：**公开 / 登录可访问 / 管理员**
- 权限校验在 middleware 完成，不散落在各路由
- 用户身份统一从 `request.state.user` 获取，禁止前端传入

### 5.3 数据安全

- 密码：{{bcrypt / argon2}} 哈希
- 敏感字段：手机号/身份证在日志中脱敏（显后4位）
- 密钥/Token 禁止进日志、禁止进响应体
- 所有用户输入在 route 层用 Pydantic 校验

## 六、API 设计规范

### 6.1 风格

- **RESTful**，资源复数：`POST /api/users`、`GET /api/users/{id}`

### 6.2 响应格式

## 八、违宪检查

> 以下检查由 AI 在每次提交代码前执行。违宪分级详见根 `CLAUDE.md §七、违宪自检`。

**提交前必查：**

- [ ] 有没有越层调用？（routes 写 SQL / services 手管连接）
- [ ] 有没有引入黑名单技术栈？
- [ ] 有没有未授权的安全绕过？（未校验用户身份、密钥硬编码）
- [ ] 有没有改了 API 格式没同步前端？（响应字段变更需要前后端同步改）
- [ ] 异常有没有暴露给用户？（500 错误应返回通用提示而非 traceback）
- [ ] 有没有报错默默修复不告知？
