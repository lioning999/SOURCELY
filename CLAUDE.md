## 铁律（不遵守就出问题）

### 一、手术式修改
- 只改必须改的，不改进相邻代码、注释、格式
- 不重构没坏的东西
- 匹配现有风格
- 无关的死代码 → 指出，不删
- 因你的改动产生的孤立代码 → 删
- **铁律一优先于铁律三：简化非本次改动的现有代码前，必须先向用户确认**

### 二、先读后写
- 改文件前先读它的当前内容
- 新增功能前先读同层现有文件结构
- 新增 API 端点前 grep 已有路由
- **禁止凭训练数据推测项目结构**

### 三、简单至上 + 过度设计判定（客观清单）
满足任意一条 = 过度设计，需简化：
```
□ 新增了需求未提及的配置项
□ 为只调用一次的逻辑创建了独立类/模块
□ 函数参数超过 5 个且部分参数当前未被使用
□ 新增的抽象层只有一个实现
□ 引入了当前需求不需要的第三方依赖
```

### 四、修改联动检查
同一个值出现在 ≥3 个文件中 → 先向用户说明，等确认后再抽取。
- 改 Python 常量 → grep 所有 import 和使用点
- 改 API 路由/响应格式 → grep 前端所有调用，确认前端同步
- 改函数签名 → grep 所有调用方
- **改判词逻辑 → 必须同步 `src/api/domain/verdict_engine.py` 和 `src/web/static/js/pages/report.js` 两处。改一处不改另一处 = bug（详见 `docs/技术-正式开发前风险清单.md` §判词双引擎同步）**

**每次代码改动完成后，执行以下 6 项检查（不允许跳过）：**
```
1. 改过的名字 → grep 全项目（含 CLAUDE.md）确认无残留旧名
2. 改过的常量 → grep 前端是否硬编码旧值
3. 改过的文件 → 确认无函数体 import、无死引用
4. 新增/删除文件 → 确认注册/解注册
5. config.py 改动 → 确认 .env.example 同步
6. 对照 `docs/违宪自检速查表.md` 自检（越层/技术栈/安全/接口格式/报错暴露）
7. 改判词规则 → grep 两端 hasCert/isAdvancedCert/isFactory/isTrader 确认一致
```

### 五、异步竞态防御
1. 创建新的 setInterval/Task 前先清理旧的
2. 达到终止条件时，先杀自己再通知外部
3. 全局可变容器（dict/list/queue）必须有上限和清理机制

### 六、修改边界
| 类型 | 能否修改 |
|------|---------|
| `CLAUDE.md` / `docs/项目架构宪法.md` | ❌ 需明确授权 |
| 项目源代码 | ✅ |
| `.claude/` / `.vscode/` | ❌ |
| 系统级配置 | ❌ |

讨论不算授权——用户说"执行""改"才算。

### 七、违宪处理
发现违宪（见铁律四第 6 项自检）后，按三级处理：

| 级别 | 判定 | 处理 |
|------|------|------|
| 一级 | 越层调用、黑名单技术栈 | 直接拒绝 |
| 二级 | 未校验身份、新建文件未注册 | 24h 修复 |
| 三级 | 函数超 50 行、命名不一致 | 提醒修正 |

### 八、多步任务中途检查点
超过 2 步 → 先列计划，每 1-2 步停下汇报，等确认后继续。

## 代码约定
- 所有函数完整类型注解；注释写 WHY 不写 WHAT
- 删函数/类/变量前 grep 全项目确认零引用
- 扩展现有文件优于新建；新建文件必须注册到入口 + 顶部一行描述用途

## 输出格式
- 结论先行；被指出错误 → 承认修正，不辩解
- 不确定时标置信度：确定 / 推测 / 存疑
- 涉及跨文件架构判断且清单无法覆盖 → 说明"需人工判断"，不给看似自信的架构结论

## 文档创建规则
1. 非单接口 CRUD → `docs/需求-{主题}.md`
2. 方案无法从代码推导 → `docs/设计-{主题}.md`
3. 创建新文档 → 更新 `docs/README.md` 索引
